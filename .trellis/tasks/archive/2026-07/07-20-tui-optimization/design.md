# TUI 综合优化设计

## 边界与顺序

父任务只持有跨子任务契约与最终集成验收，不直接承载源代码改动。按下列依赖串行完成并归档子任务：

1. sync runtime 后台化；
2. 面板异步加载与数据世代；
3. 时间窗与查询范围；
4. 样式/主题/格式化统一；
5. 按需重绘、主题快照与行窗口化；
6. 选中、排序、滚轮、spinner 与死代码处置。

每个子任务完成 focused tests、fmt、clippy 后独立提交并归档。父任务最后运行 `just ci`，按 PRD X1-X7 逐项审计，再提交性能证据与 Trellis 元数据。

## 跨任务契约

- `AppState` 是 UI 状态所有者；后台任务只能通过有界结果通道提交不可变结果。
- 数据请求携带 panel、filter、time window 与 generation；回填必须全量匹配当前请求。
- 查询连接不跨线程共享；每个后台查询打开独立 `Dashboard`。
- Overview lifetime、365 天 heatmap、sync center 与 zombie 文件扫描不受 TUI 时间窗控制，其余重载荷查询必须有窗口或明确例外。
- 默认主题、All 窗口和既有 sync 摘要是语义基线。
- 不新增生产依赖、schema、索引或 web 行为变化。

## 集成与回滚

子任务按独立提交形成回滚点。若后续任务发现前序接口不足，回到前序任务补充测试与契约，不在调用方复制第二套状态机。最终以同一数据库快照比较 All 窗口数据，并记录 release 构建下 3 次中位数性能证据。
