llmusage Sync Command Center

  • Design confirmed
  • 2026-05-29
  • Repository: D:\Documents\Code\CLI\llmusage
  • Scope: dashboard sync top area

Summary

Add a TokenTracker-inspired Sync Command Center at the top of the llmusage dashboard. The module should first answer whether it is safe to run sync, then summarize the most recent sync result. The approved baseline is a small structured sync view-model returned by the backend and rendered by the frontend, rather than a purely visual CSS pass or a large real-time analytics rebuild.

The design intentionally copies TokenTracker's strong top-level hierarchy, large metrics, segmented bars, and compact source cards, while optimizing the product meaning for llmusage: local safety, rebuild risk, worker lock state, source-level sync outcome, and clear distinction between known numbers and unavailable deltas.

Goals

  • Make the page top answer: can I safely sync now, and what should I watch before doing it?
  • Show the most recent sync outcome with structured values: status, source count, seen events, inserted delta events, stored events, timing, and source errors.
  • Keep the existing dashboard job lifecycle: start job, poll job snapshot, cancel running job, then reload the dashboard when complete.
  • Expose rebuild risk separately from ordinary sync so users understand that ordinary sync preserves imported history while rebuild may be lossy.
  • Preserve the existing Rust data path, SQLite store, SyncShard commit protocol, and dashboard API timeout boundaries.
  • Stay bilingual-ready by routing visible labels through the existing copy/i18n surfaces.

Non-goals

  • No implementation of a new parser or provider source in this design.
  • No TokenTracker queue file model; llmusage continues using SQLite and SyncShard.
  • No cloud upload, public leaderboard, account identity, social sharing, or remote telemetry.
  • No claim of per-run token or cost delta unless the backend records reliable before-and-after evidence. Until then, the honest delta is newly inserted events.
  • No destructive rebuild action as the primary call to action. Rebuild stays secondary and risk-gated.
  • No broad dashboard redesign beyond the top sync command area and its direct dependencies.

Context

The reference material came from the local ref\TokenTracker\docs\screenshots assets. The raw URL https://raw.githubusercontent.com/mm7894215/tokentracker/main/docs/screens was checked and returned 404, so the design relies on the checked-out local reference screenshots instead. The relevant screenshots show a strong top surface: huge token totals, source segmentation, compact metric cards, activity heatmaps, and a menu-bar style status panel.

The current llmusage dashboard already has the necessary primitives: src/web/shell.rs defines the dashboard shell and sync button, src/web/assets/app.js starts and polls sync jobs, src/sync/job_registry.rs stores pollable job snapshots, src/parsers/mod.rs defines sync events and per-source stats, and src/web/assets/data/derive.js normalizes health and diagnostics into page context. The new design should compose these facts instead of inventing a second sync system.

Repository ADRs reinforce the boundary. ADR 0002 keeps SyncShard as the commit protocol, and ADR 0007 requires explicit error surfaces. This spec follows both: the command center is a presentation and small API contract improvement, not a rewrite of writer semantics.

Recommended Approach

Use a small backend-owned sync_command_center view-model and render it as a dedicated top module. This is the approved second approach from the design discussion: more truthful and testable than a frontend-only composition, but much smaller than a full TokenTracker-style real-time report system.

Alternatives

Alternative Why not the baseline When it becomes useful
Frontend-only reuse of existing data Lowest risk, but it would either parse summary strings or infer per-run facts from dashboard aggregates. That weakens trust in the top answer. Useful as a temporary prototype if backend API work must be deferred.
Full TokenTracker-style real-time report Strong visual impact, but it requires durable before-and-after metrics, richer progress events, and more UI states. It is too broad for this top-area optimization. Useful after sync run logging records token and cost deltas as first-class facts.
Merge into the existing hero/status card only Good consistency, but it keeps sync as secondary content. The user selected a dedicated command-center direction. Useful if later visual review shows the page top is too tall.

Design Details

Backend view-model

The backend should return a stable structure in /api/dashboard, or through a sibling endpoint if maintaining strict separation is preferred. The field name should be explicit, such as sync_command_center. The model should be built from existing overview, health, diagnostics, source rows, and recent job/run facts. 

{
  "sync_command_center": {
    "tone": "good",
    "headline": "Sync ready",
    "reason": "Regular sync is safe; rebuild has one source risk.",
    "mode": "live",
    "current_job": {
      "job_id": "job-42",
      "status": "running",
      "last_event": "source_finished",
      "started_at": "2026-05-29T00:00:00Z",
      "finished_at": null,
      "error": null
    },
    "last_run": {
      "status": "completed",
      "finished_at": "2026-05-29T00:01:12Z",
      "sources": 4,
      "total_seen": 230,
      "total_inserted": 128,
      "stored_events": 18400
    },
    "safety": {
      "ordinary_sync_safe": true,
      "worker_lock": "available",
      "lossy_rebuild_risk": true,
      "risk_sources": ["gemini"]
    },
    "sources": [
      {
        "source": "codex",
        "status": "ok",
        "share": 0.46,
        "inserted_delta": 74,
        "stored_events": 9200,
        "last_error": null
      }
    ]
  }
}

The exact Rust type can be smaller than this example in the first implementation. The important contract is that the frontend receives structured values and never scrapes the human summary string.

Frontend rendering

  • Add a renderer such as renderSyncCommandCenter(context, dashboardState).
  • Place the host near the top of shell.rs, before filters and KPI cards, so it reads as the first operational answer.
  • Use existing theme variables and copy/i18n patterns. Keep the dark card compatible with both light and dark page themes.
  • Render five states: empty or idle, ready, running, failed, and rebuild risk.
  • Keep snapshot mode truthful: show captured status and disable live sync actions.

Action behavior

  • Run sync keeps using POST /api/jobs with current source and range options.
  • Cancel sync keeps using the current cancel endpoint when the active job is running.
  • Rebuild is not the primary action. If exposed later, it needs explicit warning text when lossy_rebuild_risk is true.
  • Auto-refresh should pause or avoid duplicate reloads while a job is running, matching existing behavior.

Risks

Risk Mitigation
The UI implies token or cost delta that the backend cannot prove. Use inserted event delta as the first version. Add token/cost delta only after run logging records before-and-after totals.
The top card becomes too tall and pushes current dashboard content down. Use a compact two-row layout at desktop sizes and collapse source cards behind a concise list on narrow screens.
Job state is in memory while dashboard totals are persisted, so after reload the active job may disappear. Show durable last-run data from persisted run logs or overview, and treat current job as optional live state.
Rebuild risk messaging scares users away from ordinary sync. Text must separate ordinary sync safety from rebuild risk: ordinary sync preserves imported history; rebuild can be lossy when sources are missing.
Frontend duplicates insight logic that already exists in derive.js. Keep safety classification in the backend view-model or one small derive helper with asset tests preventing summary-string parsing.

Test/Acceptance Criteria

  • Backend test proves /api/dashboard exposes sync_command_center with stable fields for ready, failed, running, and rebuild-risk fixtures.
  • Job registry or web route test proves completed, failed, and cancelled snapshots still serialize with stable status and structured last-event data.
  • Frontend asset test proves the renderer is wired from the shell/app entry and does not parse snapshot.summary text.
  • Browser visual check covers empty DB, normal data, running job, failed job, and lossy rebuild risk at desktop width and a narrow responsive width.
  • No horizontal overflow appears in the command center source-card row.
  • Snapshot mode shows captured information and disables live sync/cancel actions.
  • Minimum verification for implementation: targeted web/API tests plus cargo test -- --test-threads=1. If docs screenshots or docs navigation change, also run the docs build.

Review Status

Design discussion result: approved to continue with the small structured Sync Command Center contract, combining approach B with the priority order C: safety answer first, latest sync summary second.
